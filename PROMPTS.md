3. Chechout assíncrono - Vc é um desenvolvedor backend especialista, crie um feature que vai implementar um endpoint POST: v1/orders/checkout que retorn um status 202 e a orderId. O endpoint vai receber um header idempotency-key que será um uuid. No body deve ser enviado(id do produto, quantidade). O checkout antes de salvar o pedido deverá fazer um update na tabela de stocks reservando no estoque a quantidade enviada. Utilizando atomic update condicional. O logge deve registrar cada acção feita no checkout com a classe Logger instanciada no controller. caso o estoque tenha disponibilidade, salvar o pedido e enviar para fila na mesma transacao. A mensagem da fuila vai ter o correlationId que é o id da requisicao, idempotency_key, quantidade reservada do estoque e disponibilidade do estoque, orderId. criar um registro no redis com a idempotency-key com o status do pedido.
Criar um worker que vai consumir da fila. Vai pegar a idempotency-key e verificar se existe no redis e se o status do pedido. Caso for completed, retorna e pega a proxima mensagem, caso nao existe no redis cria com o status PROCESSING. Terá um log no worker simulando requisicao com sucesso. Apos isso ai realmente decrementa do estoque utilizando o update atomic condicional e atualiza o status do pedido no banco no final marca a chave idempotency-key do redis como PROCESSED
Crie a Specificação, um plano de implementação e as tasks. Não execute ate minha validação e review e só faça oq está no escopo do plano, ao final faça um code review e os testes do endpoint. Cada etapa deve ser executada em modo subagent. A medida que for fazendo as tarefas marcar o check no task.md. Todo o fluxo de checkou/worker na acoes devem utilizar a classe Logger do processo com o reqId, idempotencyId, orderId estoque e mensagem de cada açao.
